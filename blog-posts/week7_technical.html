Avoiding SQL Injections
September 21st, 2014

To begin, SQL injections are a malicious way of tampering with Databases.

An SQL injection can gain access to private information or even destroy a table.

The 'or 1=1' approach seems to be one of the most common. If the SQL statement is looking for something to equal 'true' before more information is accessed then adding 'or 1=1' can trick the program. Here is an example,

User ID: John or 1=1

If you enter John or 1=1 into a user id input then it could read it like this,

SELECT * FROM Users WHERE UserId = 105 or 1=1

As you can see, no matter what is written before the 'or 1=1', the statement will always be true. This could leave private information open for anyone to see.

Another SQL injection method is based on Batched SQL statements.

This method takes advantage of the fact that SQL separates statements with a ';'. So, if there is a space for user id input and you type,

"John; DROP TABLE Accounts"

Then, the program could read that as two separate statements and take John as the user ID and then Delete the Accounts table.

OK, so what can we do about this?

There are a few approaches we could take, but the best method is to use parameters.

From what I have read, it looks like SQL parameters are used by representing the user id as a specific symbol such as '@sql'. This way, the only code that will be run is the symbol that is given, so no actual user input will be used.

This really looks to be the only approach that people agree is effective, but I still couldn't find as much information on the topic as I would have thought. I hope that I was able to understand SQL injections and correctly explain their danger.
