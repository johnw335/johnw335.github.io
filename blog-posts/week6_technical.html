Inheritance vs Composition
September 11th, 2014

If I have learned anything from my reasearch on this topic, it's to only use Inheritance when you have to. Programmers tend to like Composition much more.

But... we still need to know how to use both.

Here is an example of Inheritance. (Here they also use super() which I will explain

Here the class "Child" inherits from "Parent". So, you could call a method that exists in child or you can call a method from child that only exists in parent.

... hmmm ... if you don't understand that, don't worry. It's my writing and not your abilty to get the concept.

super() is a ruby method that will call the inherited information and then the information within the method called.

This means that son.altered() will return

>>CHILD, BEFORE PARENT altered()

>>PARENT altered()

>>CHILD, AFTER PARENT altered()

Ok, before I get too stuck talking about Inheritance, we should talk a little about composition.

Composition is a way of accomplishing the same goals without having to worry about the mess of inheritance. Inheritance works by just calling the functions you need within your class.

We could use Composition in the example above by simply saying

@parent = Parent.new

while initilizing Child

this way we can call methods from parent whenver we need them.

I know this has been a simplistic explanation of the topic, but I hope you have taken something useful away from it. It seems like there is a time and place for each approach, but Composition is cleaner.
